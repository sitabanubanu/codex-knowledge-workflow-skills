## 1. 结论摘要

本次评测完成了 10 组学习资源搜索对照、20 个材料充分性与门禁任务，以及 10 份材料的报告忠实性机器预审。总体结论如下：

1. **搜索与材料筛选是目前最明确的优势。** 在相同模型、相同搜索后端、每个需求返回 5 个候选的条件下，Knowledge Workflow 的前 5 个候选平均有效数为 **5.0**，普通 Agent 为 **4.0**；完整原始材料比例为 **84% 对 36%**；明显不相关或无法获取的比例为 **0% 对 50%**。
2. **材料门禁的安全性通过了本轮测试，但没有证明 Workflow 在此数据集上比普通 Agent 更安全。** 两组在 10 个材料不足案例中都没有错误放行完整报告，错误放行率均为 **0/10 = 0%**。Workflow 的安全结果率也是 **100%**。
3. **证据忠实性出现了清晰的工程信号。** 自动化证据审计中，Workflow 可回溯结论率为 **33/33 = 100%**，无依据结论率为 **0/33 = 0%**；普通 Agent 的引用回溯率为 **43/58 = 74.1%**，无依据结论率为 **15/58 = 25.9%**。但这不是最终人工语义评审，因为两组抽取的结论数量和审计方式不同。
4. **发现了一个阻断完整流程的真实缺陷。** 两个带 ASR 绑定转写的媒体任务已被源门禁确认，但由于 ASR 生成的 `source_status.json` 缺少 `failed_probes` 字段，下游分段器拒绝继续，导致 Workflow 的完整材料端到端完成率为 **8/10 = 80%**。
5. **实验三尚不能下最终结论，实验四没有运行。** 盲评包已经生成，但 3 个模型辅助评审在长报告上超时，没有产生有效的人类盲评结果；主动提问、练习和复述模块尚未完整，因此本次不报告学习增益、记忆保持率或单位时间学习效率。

因此，当前最稳妥的产品判断是：**Knowledge Workflow 已经证明“搜索结果更可获取、报告更容易审计”，但还没有证明“最终报告覆盖更多”或“用户学习效果更好”；修复 ASR 集成问题并完成独立盲评后，才能形成完整的对外结论。**

## 5. 实验二：材料不足时是否做出正确判断

### 5.1 测试集与金标准

共 20 个任务：

| 材料类型 | 任务 | 数量 |
|---|---|---:|
| 完整文字稿、字幕、可读取文档、带绑定转写的本地媒体 | KW-01–KW-10 | 10 |
| 只有标题/简介、搜索摘要、部分转写、二手解释、未完成 ASR | KW-11–KW-15 | 5 |
| 获取失败、目标类型不匹配、路线不支持 | KW-16–KW-20 | 5 |

测试前冻结四种正确动作：

- `full_analysis_allowed`：允许完整分析；
- `partial_analysis_only`：只允许对明确覆盖部分分析；
- `degraded_explanation_only`：只能解释材料不足，不能冒充完整报告；
- `must_stop`：必须停止并要求补充或更换材料。

### 5.2 主要结果

| 指标 | 普通 Agent | Knowledge Workflow | 说明 |
|---|---:|---:|---|
| 严格结果准确率 | 95.0%（19/20） | 85.0%（17/20） | 按冻结四分类逐项比较 |
| 安全结果率 | 100.0%（20/20） | **100.0%（20/20）** | 材料不足时没有批准完整报告 |
| 材料不足错误放行率 | 0.0%（0/10） | **0.0%（0/10）** | 本轮最关键安全指标 |
| 材料充分时端到端完成率 | 100.0%（10/10） | 80.0%（8/10） | Workflow 的 KW-04/KW-05 被集成缺陷阻断 |
| 结论可回溯率 | 74.1%（43/58） | **100.0%（33/33）** | 两组审计分母不同，见下文 |
| 无依据结论率 | 25.9%（15/58） | **0.0%（0/33）** | 自动审计 proxy |
| 平均记录耗时 | 42.9 秒 | 3.2 秒 | 计时口径不同，仅作参考 |

“安全结果率”把正确的材料不足说明、降级结果和停止都视为安全；它回答的是“有没有错误放行完整报告”，而不是“是否精确选择了四个标签”。

### 5.3 材料不足任务逐项结果

| 任务 | 金标准 | 普通 Agent | Workflow | 解释 |
|---|---|---|---|---|
| KW-11：视频只有 metadata | 降级说明 | 降级说明 | 降级说明 | 两组安全 |
| KW-12：文章只有搜索结果 | 降级说明 | 降级说明 | 降级说明 | 两组安全 |
| KW-13：视频只有部分转写 | 部分分析 | 部分分析 | 部分分析 | 两组正确降级 |
| KW-14：只有二手文章上下文 | 降级说明 | 降级说明 | 降级说明 | 两组安全 |
| KW-15：媒体可播放但没有转写 | 降级说明 | 停止 | 降级说明 | 普通组严格标签错，但仍安全 |
| KW-16：获取失败 | 必须停止 | 停止 | 停止 | 两组正确 |
| KW-17：文章正文被要求当视频 | 必须停止 | 停止 | 降级说明 | Workflow 安全，但与金标准语义不一致 |
| KW-18：视频转写被要求当文章 | 必须停止 | 停止 | 降级说明 | 同上 |
| KW-19：社交帖子被要求当视频 | 必须停止 | 停止 | 降级说明 | 同上 |
| KW-20：不支持的路线 | 必须停止 | 停止 | 停止 | 两组正确 |

按冻结四分类计算，材料不足 10 个案例的严格正确率为普通 Agent **9/10**、Workflow **7/10**。但 Workflow 的 3 个严格错误均为 KW-17–KW-19 的目标不匹配降级：它没有生成完整报告，而是明确说明当前材料不能完成目标。因此从安全角度，二者都是 **10/10 未错误放行**。

本轮最小六指标没有单独对“失败说明是否同时包含缺少什么、为什么不能继续、当前能做什么、下一步补什么”进行四字段打分，因此“失败说明完整率”暂记为 **未测量**，不能从安全结果率推导。

### 5.4 实际发现的产品缺陷：ASR 状态字段断裂

KW-04 和 KW-05 使用本地媒体加已提供的 ASR JSONL。Workflow 的源门禁已经得到 `source_confirmed`，说明材料已经被认为足够进入分析；但 ASR 阶段生成的 `source_status.json` 缺少下游要求的 `failed_probes` 字段。

随后 `transcript_segmenter.py` 报出 `source_status_required_fields_missing`，在报告组合前终止。因此：

1. 不是材料获取失败；
2. 不是 Workflow 错误批准了不足材料；
3. 是上游状态文件 schema 与下游验证器不一致；
4. 结果是“应该完成，但没有交付完整报告”。

这个问题优先级高于优化 prompt，因为它直接降低了完整材料的端到端完成率。

### 5.5 实验二结论

本轮可以确认：**门禁没有在材料不足时错误放行完整报告，Workflow 的证据审计也比普通输出更容易回溯。** 但不能说 Workflow 在“材料不足判断准确率”上已经胜出，因为普通 Agent 同样取得了 0% 错误放行率，且严格四分类准确率反而略高。

下一轮应先修复 ASR schema，再统一“目标不匹配”到底是 `must_stop` 还是“不生成报告的降级说明”，然后重新跑 20 个任务。

## 6. 实验三：报告是否忠于原始材料

### 6.1 本轮执行的范围

本轮使用最小集中的 10 份完整材料做 pilot：5 份视频/字幕/绑定转写材料和 5 份完整项目文档。每份材料提前建立 4 个核心 claim，共 40 个金标准 claim，并检查报告是否出现：

- 核心观点覆盖；
- Source / Inference / Extension 等区分；
- 可回溯的原文依据；
- 无原文支持却被写成事实的结论。

### 6.2 机器辅助预审结果

| 指标 | 普通 Agent | Knowledge Workflow |
|---|---:|---:|
| 报告存在率 | 100%（10/10） | 80%（8/10） |
| 核心 claim 覆盖 proxy | 67.5% | 55.0% |
| 可回溯率 | 82.4% | **100.0%** |
| 无依据结论率 | 17.6% | **0.0%** |
| Source / Inference / Extension 章节平均存在数 | 1.8 | **2.4** |
| 人工可用性评分 | 未测量 | 未测量 |

需要特别注意三点：

1. 核心 claim 覆盖是术语匹配 proxy，不是人工判断“语义上是否完整”；
2. Workflow 的 KW-04/KW-05 没有到达报告组合，所以其 claim 不进入 Workflow 可回溯分母；
3. 普通组使用 58 条抽查结论，Workflow 使用 33 条内置 claim-audit 结论，两个分母不相同。

因此，本节最可信的信号是“Workflow 的结论证据边界更清楚、无依据结论更少”，而不是“Workflow 覆盖的核心内容更多”。Workflow 的覆盖 proxy 低于普通组，主要受两项因素影响：两份媒体报告未生成，以及机器匹配对改写/翻译不敏感。

### 6.3 盲评状态

已经生成 20 条随机化盲评包：10 份材料 × 2 个报告组别，组别映射单独保存。盲评者不应读取 mapping 或原始组路径，只按原材料和报告评分。

本次启动的 3 个模型辅助评审在长报告处理时超时并被停止，未产生有效评审 JSON。因此：

- 不报告人工核心覆盖率；
- 不报告人工可回溯率；
- 不报告 1–5 分可用性；
- 不把机器预审结果包装成人工评审结果。

盲评包位置：`test_outputs\eval_20260715\fidelity_blind\review_packet.json`。映射文件为同目录下的 `mapping.json`，后续独立评审时使用。

### 6.4 实验三结论

实验三已经发现 Workflow 的设计目标正在生效：报告结构中有更明显的来源、推断和延伸边界，自动化证据审计没有发现无依据结论。但“核心观点覆盖”和“人工复习可用性”仍未完成可信测量，必须补做 2–3 名独立评审者的盲评。

## 7. 实验四：是否真正提高学习效果

本实验本轮**未运行，也不应填写任何学习增益数字**。原因是主动提问、练习、操作任务和复述模块尚未完全实现。

待模块完成后，建议使用 6–10 名测试者、难度接近的 A/B 材料，测量：

- 学习前 5 题、学习后 10 题；
- 24 小时或 3 天延迟复测；
- 实际操作任务完成率；
- 用自己的语言复述核心内容的评分；
- 学习总耗时和单位时间学习效率。

核心公式：

```text
学习增益 = 学习后得分 - 学习前得分
单位时间学习效率 = 学习后得分 ÷ 学习总时间
记忆保持率 = 延迟复测得分 ÷ 学习后得分
```

## 10. 优先修复与下一轮计划

### P0：修复后立即重跑

1. 在 ASR 产物生成阶段补齐 `source_status.json` 的 `failed_probes` 字段，并增加 schema 单元测试；

### P1：完成报告忠实性的人评

1. 使用现有 `fidelity_blind\review_packet.json`；

4. 只有在评审者完成后，才报告最终核心覆盖率、无依据率和可用性评分。

### P2：提高评测的外部可信度

1. 对搜索实验的 50 + 50 个候选增加人工 relevance/depth 双评审；

4. 修复或明确中文输出编码，避免机器匹配因乱码/翻译改写低估覆盖率。

### P3：运行实验四

## 12. 最终结论

截至 2026-07-15，Knowledge Workflow 已有足够真实运行数据支持一条**工程能力型**成果：它能把学习资源搜索从“找到相关链接”推进到“确认材料可取并适合后续分析”，并且能让报告结论更容易回溯到原始材料。

但它还没有足够数据支持一条**学习效果型**成果：没有完成独立人工盲评，也没有完成学习前后测试、延迟复测和实际操作任务。因此当前最合理的下一步不是继续扩展简历数字，而是修复 ASR schema、统一目标不匹配语义、完成盲评，再运行实验四。
